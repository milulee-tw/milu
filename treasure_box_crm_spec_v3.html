01 提案概要與目的 OVERVIEW & PURPOSE
核心假設 · HYPOTHESIS
透過「累積押注金額解鎖寶箱」的漸進式獎勵機制，驅動玩家在當日內持續押注，同時以隔日重置的急迫感製造次日回流動機，達到提升單日 Spin 數與 D1 留存率的雙重目標。
🎯
主要目標
提升新手玩家單日押注總量，透過累積門檻設計讓玩家有持續押注的具體動機
🔄
次要目標
隔日重置機制製造跨日懸念，今日未翻完的進度清零，驅動玩家次日回流繼續
📊
驗證指標
D1 留存率變化、活動組 vs 對照組單日 Spin 數差異、活動參與率、完整循環完成率
背景問題點
⚡
單日押注動力不足
玩家除日常消費外，缺乏在單日內集中押注的明確誘因
📅
次日回流率偏低
玩家無未完成任務牽引時，次日不回流的機率顯著提升
🆕
新手黏著度低
新進玩家前期缺乏低門檻回饋，習慣尚未建立前容易流失
功能定位：本次切版核心原則
玩家面 v1 只發現金獎，但後端基建一次做滿、AWC API 一次串完。v1 的 place_bet 與 v2 才會用到的 LoginWithoutRedirect 等需求一次性打包提交給 AWC 集成商，避免分兩次串接造成重複的開發、溝通與等待成本。FREE SPIN 卡的「水管」（資料表 + AWC API + max win 上限設定）在 v1 就建好，實際發卡與使用體驗留到 v2。

解鎖門檻設計（累積押注金額）
為什麼改用累積押注金額而非 Spin 次數？
Spin 次數對不同押注段的玩家體驗差異極大——低押注玩家每次 ₱1，高押注玩家每次 ₱100，同樣的 Spin 數對後者幾乎沒有成本感。
改用押注金額後，大 R 玩家可因高押注段快速過關，機制對各押注段玩家更為公平且有感。
增量門檻 vs 內部累積：下表 [50, 100, 150, 200, 300, 400, 600, 800, 1,000] 為「每格增量門檻」。內部 today_wager 持續累加，解鎖第 d 格的條件為 today_wager ≥ 累積門檻[d]，其中累積門檻 = 增量 running sum = [50, 150, 300, 500, 800, 1,200, 1,800, 2,600, 3,600]，完整循環內部累積總押注 = ₱3,600。

UI 顯示原則：玩家畫面只顯示「再押 ₱X 解鎖下一格」（X = 下一格累積門檻 − today_wager），進度條呈現「本階段進度」；永遠不顯示 ₱3,600 累積總門檻，讓玩家覺得每階段都好達成（詳見第 06 章 UI）。

重置規則
🕛
每日 00:00 重置
當日累積押注（today_wager）清零，獎項組合重新產生並重新洗牌。最終獎位置每日動態隨機，不再固定於特定格
⏸️
當日未翻完不補
今日累積未達門檻的格子，隔日直接清零。不保留進度——製造急迫感驅動玩家當日完成
🎲
動態洗牌分配
每日開始時執行洗牌，小獎×5、中獎×2、大獎×1、最終獎×1 隨機分配到9格。玩家看到攤開結果後點「開始遊戲」才執行洗牌，格子翻回背面。
玩家路徑分析
v2 改為動態洗牌機制，最終獎位置每日隨機，不再固定於第 7/8/9 格。玩家每次的最終獎解鎖門檻取決於洗牌後最終獎落在第幾格，增加每日新鮮感與公平性。
03 獎勵設計與 RTP 計算 PRIZE STRUCTURE & RTP
獎勵結構（9 宮格組成）
💙 ₱1
小獎 ×5格
💚 ₱3
中獎 ×2格
🌟 ₱8
大獎 ×1格
👑 ₱18
最終獎 ×1格（原 Jackpot）
合計 9 格，每日洗牌後隨機分配位置。
FREE SPIN 卡（v2 新增獎勵類型）
💳
道具性質
FREE SPIN 卡為道具類獎勵，不直接發放現金，而是存入玩家背包，可自行選擇時機使用
🎮
使用方式
玩家進入背包後點擊 FREE SPIN 卡，選擇允許使用的 Combo 遊戲，系統跳轉進入該遊戲並自動套用免費局數
⚙️
技術機制
後台發卡時設定允許的 GameId 清單與局數。跳轉時呼叫 AWC LoginWithoutRedirect API，取得一次性遊戲入口 URL 後跳轉
RTP 計算驗證
固定獎池模型：獎池固定 = 小獎 ×5(₱1) + 中獎 ×2(₱3) + 大獎 ×1(₱8) + 最終獎 ×1(₱18) = ₱37。

解鎖門檻（每格增量）：₱50 / ₱100 / ₱150 / ₱200 / ₱300 / ₱400 / ₱600 / ₱800 / ₱1,000，內部累積總押注 = ₱3,600。

完整循環 RTP = ₱37 ÷ ₱3,600 = 1.028%（容許區間 0.9%–1.1%，✓ 通過）

翻完 9 格的人必得 ₱37，RTP 恆為 1.028%；逐抽權重的作用是「把大獎／最終獎壓到後段」，控制早退玩家的 RTP 暴露。

安全設計：洗牌結果解鎖時即寫死
洗牌結果（各格獎項內容、最終獎落點）在玩家當日第一次解鎖時即決定並寫入資料庫，後續不允許重抽；玩家無法透過重整或重新登入改變結果，避免作弊風險。累積門檻為每格增量、UI 僅顯示下一階段所需，不顯示 ₱3,600 總額（見第 06 章 UI）。

玩家押注後的系統處理流程
1 · AWC CALLBACK
收到 place_bet 回調
AWC 在每次押注後回傳 game_account、bet_amount、game_code
↓
2 · PROGRESS UPDATE
累積押注金額至玩家進度表
UPDATE player_progress SET today_wager += bet_amount WHERE player_id AND campaign_id AND date = today
↓
3 · THRESHOLD CHECK
檢查是否達下一格門檻
判斷 today_wager ≥ 下一格累積門檻 → 若第一次達到：執行洗牌寫入（若尚未）→ 標記該格為 unlocked
↓
4 · PUSH NOTIFICATION
通知前端格子已解鎖
回傳解鎖狀態，前端顯示格子發光動畫提示玩家可翻開
玩家點擊翻格後的獎勵派發流程
1 · CLICK EVENT
玩家點擊已解鎖格子
前端送出翻格請求（player_id, campaign_id, cell_index）
↓
2 · VALIDATE
後端驗證狀態合法性
確認：① 格子狀態為 unlocked ② 今日尚未翻過 ③ 非重複請求
↓
3 · GRANT REWARD
呼叫 AWC place_bet 派發現金
bet_amount=0，標記 bonus_type=CHEST_REWARD, campaign_id。等待 AWC 回成功後才更新狀態
↓
4 · RECORD & UPDATE
寫入發放紀錄，更新格子狀態
INSERT RewardGrant（active）→ UPDATE cell_state = opened → 回傳獎勵金額給前端
↓
5 · FAILURE HANDLING
AWC 失敗時的處理
RewardGrant 維持 pending 狀態，排隊重試。格子不標為 opened，玩家不顯示獎勵，避免發出玩家用不到的獎
每日重置排程
⏰ 觸發時機
每日 00:00 伺服器時間（菲律賓 UTC+8）自動排程觸發
🔄 重置內容
清除玩家當日 today_wager、格子狀態（unlocked/opened → locked）、洗牌結果
📝 保留內容
歷史發放紀錄（RewardGrant）全保留，供對帳使用。不影響玩家歷史派彩

3.6冪等性保護🛠 後端同一格重複翻格請求（前端重傳）需被正確攔截，不重複派獎
3.7每日重置排程🛠 後端每日 00:00 UTC+8 清空當日 player_progress，不清歷史 reward_grant⚠️ 基礎建設
3.8查詢玩家寶箱狀態 API（GET /chest/status）🛠 後端回傳當日進度、各格解鎖狀態、已翻格獎勵，供前端渲染
3.9FREE SPIN 卡派發邏輯🛠 後端翻格結果為 FREE SPIN 卡時，寫入 inventory 資料表，不呼叫 AWC place_bet，改以道具形式儲存（實際發卡）v2 啟用
3.10背包查詢 API（GET /inventory）🛠 後端回傳玩家目前持有道具清單，含道具類型、數量、可使用遊戲列表、有效期限v2 啟用
3.11背包使用 API（POST /inventory/use，道具使用跳轉）🛠 後端驗證道具有效 → 呼叫 AWC LoginWithoutRedirect（帶 Token + GameId）→ 取得一次性 URL → 回傳前端 → 更新 item_use_log（AWC 已於 v1 串妥）v2 啟用
