01
系統目標
這個系統要解決什麼問題，提供哪些核心能力

遊戲管理後台需要一套可讓指定管理員彈性調整遊戲出獎率（RTP）的機制，並支援機率版本的 A/B 測試，以透過數據驗證不同機率配置對玩家留存的影響。

系統必須支援以下能力
  • 多維度 RTP 差異化設定：可依線路、幣值、遊戲、玩家群組等維度組合設定不同 RTP 數值，條件越精確優先序越高
  • 規則有效期管理：每條 RTP 規則有明確的生效與失效時間，時間到期後自動移入已結束狀態
  • 群組與特殊事件：可將特定玩家帳號歸入群組，並對群組設定特殊遊戲事件（如強制進入 FreeGame）
  • 機率 A/B 測試：同一款遊戲可同時運行兩套機率配置進行對照實驗，透過玩家分群追蹤各組留存數據，協助決策最佳機率版本
  • 族群差異化體驗：A/B 測試確定基礎版本後，可對新手、高 RTP、低 RTP、特殊照顧等不同玩家族群施加差異化機率調整
  • 完整操作紀錄：所有 RTP 設定變更和 A/B 測試操作均記錄於審計日誌，不可刪除
  • RTP 模擬器：可在不實際生效的情況下，模擬查詢特定條件下會命中哪條規則及 RTP 數值
適用範圍
項目說明

優先級High — 涉及合規、財務與玩家體驗

03
RTP 優先序機制
多條規則同時命中時，系統如何決定使用哪一條

系統維護一份全局優先序排名。當玩家發起遊戲請求時，系統找出所有條件命中的進行中規則，執行排名最高（數字最小）的那一條。若無任何規則命中，依序 fallback 至線路預設值，再到全局預設值。

優先序排名規則
  • 排名為全局唯一整數，數字越小代表優先序越高，由 Admin 在建立規則時手動指定插入位置
  • 若新規則要插入的位置已被占用，系統顯示受影響規則清單，Admin 確認後才執行批次往後位移
  • 條件組合越精確（越多欄位有值），建議排在越前面；但最終排名由 Admin 決定，系統不強制
條件維度與建議排名參考
建議區間條件組合說明
最高群組 + 遊戲 + 線路 + 幣值最精確，鎖定特定群組且指定遊戲/線路/幣值
↓群組 + 遊戲針對特定群組在特定遊戲
↓群組針對特定群組全部遊戲
↓線路 + 幣值 + 遊戲針對特定線路/幣值/遊戲，不限群組
↓線路 + 幣值針對特定線路/幣值
↓線路針對特定線路全部遊戲
↓遊戲針對特定遊戲全線路
最低（無規則命中）fallback 至線路預設值 → 全局預設值
Fallback 查找順序
1
命中進行中規則
找出所有條件符合且在有效期間內的規則，取排名最高（數字最小）的一條，採用其 RTP 值
2
無規則命中 → 線路預設值
依據玩家所在線路，使用各線路設定的預設 RTP（SEA 系列 96.7%、GLOBAL 系列 95.5%）
3
線路無設定 → 全局預設值
系統設定的最終 fallback 值，確保任何情況下都有 RTP 可用
04
RTP 規則操作需求
後台介面上 Admin 需要能做到的事
規則列表頁面需求
  • 分三個標籤頁顯示：「進行中」（有效規則，按優先序排列）、「已結束」（到期或停用）、「審計日誌」
  • 篩選功能：進行中標籤頁支援依線路、幣值、遊戲、群組 ID 篩選；已結束支援依遊戲名稱與結束日期區間篩選
  • 列表欄位：排名、條件摘要（只顯示有填的欄位）、RTP 值（百分比格式，≥98% 顯示橘色警示）、生效起始與截止、操作按鈕
  • 已結束規則：僅供查閱，不提供任何編輯或重啟操作
新增 / 編輯規則需求
欄位必填說明與限制
插入排名位置是正整數，若位置已被占用則觸發 Shift 確認流程
線路否從系統線路清單選擇，留空表示不限線路
幣值否ISO 4217 代碼，留空表示不限幣值
遊戲否從遊戲清單選擇，留空表示不限遊戲
群組 ID否從已建立的群組中選擇，留空表示不指定群組
RTP 值（%）是80.00% ～ 99.00%，小數後 2 位；≥ 98% 需輸入確認碼 CONFIRM
生效起始是不可早於當前時間
生效截止否留空表示永久有效；若填寫需晚於生效起始
備註否上限 200 字，說明此規則用途
Priority Shift 確認流程需求
  • 當插入排名位置已被占用，儲存前需彈出確認視窗
  • 確認視窗需列出「將被往後推移」的所有規則，顯示每條的 ID、條件摘要、原排名 → 新排名
  • Admin 可選擇「確認執行」或「取消並重新填寫排名」
  • 確認後系統批次更新所有受影響規則，並記錄於審計日誌
停用規則需求
  • 進行中規則提供「停用」操作，點擊後需彈出確認視窗
  • 停用後規則立即失效，移入「已結束」標籤，不可重新啟用
  • 需恢復效果時，必須建立新規則
RTP 模擬器需求
  • 提供獨立的模擬器入口（頁面工具列按鈕）
  • Admin 輸入：線路、幣值、遊戲、群組 ID（選填）
  • 系統回傳：命中的規則 ID 與條件摘要、最終 RTP 值、fallback 路徑說明
  • 模擬結果不寫入資料庫，不影響線上運作
05
線路預設值管理
無規則命中時各線路的 fallback RTP
  • 獨立子頁面：「線路預設值設定」為 RTP 設定模組的第二個子頁，與活動規則分開管理
  • 列表顯示：以表格列出所有線路，顯示線路代碼、系列（SEA / GLOBAL）、目前預設 RTP、最後修改人與時間
  • 行內編輯：點擊任一線路的「編輯」可在同一列直接修改 RTP 值，儲存前需確認
  • 修改限制：RTP 值範圍同活動規則（80.00% ～ 99.00%）
  • 目前預設值：SEA 系列線路預設 96.7%，GLOBAL 系列線路預設 95.5%
  • 修改記錄：每次修改均記錄於審計日誌
06
群組管理需求
玩家帳號的群組化管理與特殊事件設定

群組是獨立管理的實體，用於將特定玩家帳號集合起來，以便在 RTP 規則中以群組為條件進行差異化設定，並可掛載特殊遊戲事件。

群組管理功能需求
功能需求說明
群組列表顯示群組 ID、備註、帳號數量、掛載的特殊事件摘要
新增群組填入群組 ID（唯一，建立後不可修改）、備註說明、帳號成員、特殊事件清單
編輯群組可修改備註、帳號成員、特殊事件；群組 ID 建立後不可更改
帳號成員輸入支援換行或逗號分隔批次貼入多個帳號；同一帳號可同時屬於多個群組
刪除群組不提供刪除功能，僅可將所有特殊事件停用；若群組已被 RTP 規則引用，需先停用相關規則
特殊事件（Event）需求
  • 一個群組可掛載多個事件，各事件獨立開關，互不影響
  • 目前支援的事件類型：強制進入 FreeGame（force_fg），可設定手數上限、觸發率、買入類型、下注倍率
  • 事件類型預留擴充空間，後續可新增其他類型（如加速 RTP）不影響現有事件
  • 移除事件為不可逆操作，需要二次確認
ℹ
同一玩家帳號可能同時屬於多個群組，多個群組對同一玩家均有 RTP 規則命中時，依優先序模型取排名最高的規則執行。
AB1
A/B Testing — 兩階段執行規劃
機率版本對照實驗的整體執行策略

機率 A/B Testing 讓同一款遊戲同時運行兩套機率配置（版本 A / 版本 B），透過追蹤真實玩家數據找出最佳機率版本。整個流程分為兩個明確階段執行。

Phase 2
族群體驗優化期
Phase 1 結束後 → 持續運行
在已確定的最佳基礎機率版本上，針對不同玩家族群（新手、高 RTP、低 RTP、特殊照顧）施加差異化微調，提升各族群的遊戲體驗與留存率。

族群 Tag 可於每次遊玩 Session 結束後更新

Phase 1 → Phase 2 銜接條件
  • 樣本門檻：A / B 兩組各需達到至少 1,000 名玩家，方可進行統計分析
  • 顯著性門檻：D7 留存率差異需達統計顯著（p-value < 0.05）才視為有明確結論
  • 不顯著時的處置：系統顯示警告，Admin 需輸入確認碼才可強制結束；建議延長測試或重新設計版本差異
  • 切換操作：Admin 在後台「結束測試」後，系統自動記錄勝出版本為基礎版本，並切換 Phase 2 分群模式；切換應在低峰時段執行，進行中的遊玩 Session 不受影響
⚠
Peeking 問題：Phase 1 測試期間，即使看到某組數據暫時領先，也不可提前結束。在達到 30 天及 1,000 樣本前做出決策，統計結論的可靠性會大幅降低。系統需在視覺上提醒 Admin 目前測試進度是否已達決策條件。
AB2
Tag 標籤定義
玩家分群標籤的種類、格式與各族群的識別條件

Tag 的有效範圍是「玩家 + 遊戲 + 線路」的組合。由於玩家只會在單一線路中遊玩，同一款遊戲在不同線路的 A/B Testing 完全獨立——不同線路的玩家有各自獨立的 Tag，彼此不共用。Tag 格式為 族群名稱 + 組別（如 base_A、newbie_B），每個月重置一次，整月有效。

A 組　base_A
User ID 尾數為奇數的玩家
套用機率版本 A
B 組　base_B
User ID 尾數為偶數的玩家
套用機率版本 B
ℹ
User ID 尾數固定，同一玩家每月會分到相同組別。Tag 每月 1 日重置，玩家該月首次在該線路進入遊戲時寫入，整月有效不再改變。

同一款遊戲在不同線路各自有獨立的 A/B Testing，不同線路的 A 組、B 組互不相關，統計時也是獨立計算。

AB3
分群運作規則
每次 Spin 的完整判斷鏈、Tag 寫入與更新時機、白名單排除機制
每次 Spin 的完整查詢優先序

玩家每次發起 Spin，系統依下列固定順序逐層判斷，每一層有結果就停止，不繼續往下：

1
白名單特殊帳號檢查
判斷此玩家是否屬於設有特殊 RTP 的白名單群組（如 50% 特殊機率群組）。
→ 是：直接套用白名單 RTP 規則，不進入 A/B Testing，流程結束
→ 否：繼續往下
2
RTP 規則命中（Priority 模型）
依玩家所在線路，查找所有進行中且條件命中的 RTP 規則，取優先序最高的一條。
→ 有命中：取該規則的 RTP 數值（如 AWS081 線路命中 98%）
→ 無命中：使用線路預設值（如 96.7%）→ 再 fallback 至全局預設值
3
A/B Testing 分群查詢
查詢「此玩家 + 此遊戲 + 此線路」當月是否有進行中的 A/B Testing。
→ 無 A/B Testing：使用預設機率配置，流程結束
→ 有 A/B Testing：查詢玩家的 segment_tag，取對應的機率版本（版本 A 或版本 B）
✓
遊戲引擎執行
將上方兩個結果合併送入遊戲引擎：RTP 數值（由步驟 2 決定）+ 機率版本（由步驟 3 決定）→ 執行本次 Spin
⚠
白名單排除是 A/B Testing 數據純淨的關鍵：白名單玩家的 RTP 已被人為介入（如 50%），其遊玩行為不代表機率版本本身的效果。若納入 A/B 統計樣本，會污染留存率數據、導致錯誤結論。系統需確保白名單玩家的 Spin 數據不計入任何 A/B 指標計算。
案例說明：遊戲 A 跨線路的實際運作
玩家類型所在線路步驟 1 白名單步驟 2 RTP步驟 3 A/B最終執行
一般玩家甲 AWS081 否 → 繼續 命中 98% 規則 有 A/B → 查 Tag → base_A 98% + 版本 A
一般玩家乙 AWS081 否 → 繼續 命中 98% 規則 有 A/B → 查 Tag → base_B 98% + 版本 B
一般玩家丙 其他線路 否 → 繼續 線路預設 96.7% 有 A/B → 查 Tag → base_A 96.7% + 版本 A
一般玩家丁 其他線路 否 → 繼續 線路預設 96.7% 有 A/B → 查 Tag → base_B 96.7% + 版本 B
白名單玩家戊 AWS081 是 → 結束 白名單規則 50% 不進入 50%（無 A/B）
ℹ
AWS081 的 A/B Testing 和其他線路的 A/B Testing 是兩個完全獨立的測試。玩家甲乙在 AWS081 的 A/B 分組，與玩家丙丁在其他線路的 A/B 分組，各自獨立計算、各自獨立決策，互不干擾。
A/B Testing 有效樣本定義

並非所有玩家的數據都可以納入 A/B 統計。下列情況的玩家或 Spin 需排除於指標計算之外：

排除情況原因
當次 Spin 命中白名單特殊 RTP 規則的玩家RTP 被人為介入，行為不代表機率版本效果
當次 Spin 命中任何非線路預設的 RTP 規則的玩家RTP 數值受其他變數影響，A/B 版本不是唯一變數
當月 Tag 切換過的玩家，切換前的歷史 Spin該段 Spin 對應的機率版本與當前 Tag 不同，歸因混亂
⚠
排除規則若設計過嚴，可能導致有效樣本數不足、測試週期被迫延長。建議在後台參數設定中，允許 Admin 選擇排除範圍（例如只排除白名單，或排除所有非預設 RTP 規則命中的玩家），並在指標監控頁面顯示「有效樣本數」與「總遊玩人數」的對比。
Tag 寫入與更新時機
  • Phase 1 寫入：玩家當月首次在「該線路 + 該遊戲」進入時，依 User ID 單雙號判斷並寫入 base_A 或 base_B，整月不變
  • Phase 2 寫入：玩家進入時，系統依族群優先級判斷適用的 Tag；若尚無本月 Tag 則立即寫入；若已有但偵測到族群條件變化（且本月尚未切換過），標記為待切換狀態
  • 待切換的生效時機：在下一次 Session（遊玩段落）結束後才正式套用新 Tag，不在遊玩中途切換，確保單次 Session 的機率版本一致
  • 每月切換上限：同一玩家在同一款遊戲同一條線路同一個月內，最多切換一次族群 Tag，確保當月指標歸因乾淨

結束測試的操作需求
  • 正常結束（條件達成）：樣本 ≥ 1,000 且測試 ≥ 30 天且 p < 0.05，顯示指標摘要後由 Admin 確認結束，系統自動記錄勝出版本
  • 強制結束（條件未達成）：若任一條件未達成，顯示警告並列出未滿足的條件，Admin 需輸入確認碼 CONFIRM 才可強制結束
  • 結束後：測試移至「已結束」列表，歷史指標數據保留供查閱，不可重新啟動（需建立新測試）

  • 高 RTP 觸發閾值：預設 98%，可依遊戲類型調整（高波動遊戲正常 RTP 分布較寬，閾值應設更高）
  • 低 RTP 觸發閾值：預設 85%，可依遊戲調整
  • 新手天數定義：預設帳號建立後 7 天內為新手，可調整為 3 天或 14 天
  • RTP Rolling 窗口：計算 RTP 所使用的近期 Spin 數量範圍，預設值由開發方建議

其他限制規則
  • 不提供批次操作：每次僅能編輯單條規則，降低誤操作影響範圍
  • 不可刪除：規則和日誌均只能停用或封存，不可物理刪除，保留歷史可追溯性
  • 停用不可重啟：已停用的規則移入「已結束」後不可重新啟用，需建立新規則

尚未確認的需求，需要各方決策
OPEN · 01 — PM + Tech Lead

全局預設 RTP 由誰設定、在哪裡管理？

建議：統一在 RTP 設定模組管理，以最低優先序的特殊規則形式存在，避免設定分散。

OPEN · 02 — PM + Backend

群組 Event 類型後續擴充方式？

新增 Event 類型是否需要工程部署，或可由後台動態新增？若可動態新增，需定義欄位驗證機制。

OPEN · 03 — PM + Compliance

RTP 變更是否需要 Maker-Checker 雙人審核流程？

若合規要求高則建議導入，但會增加操作複雜度，需評估實際監管要求。

OPEN · 04 — PM + Backend

模擬器是否需要支援「查詢歷史時間點的命中結果」？

對審計有價值，但需要快照機制，增加後端儲存成本，需評估優先級。

OPEN · 05 — PM + Data Analyst

Phase 1 測試 30 天後仍無統計顯著差異時的處置策略？

建議：最多延長至 60 天，若仍無顯著差異則判定版本無差異，以 RTP 較低的版本作為基礎版本。

OPEN · 06 — PM + Game Designer

高/低 RTP 閾值是否需依遊戲類型分別設定？

不同類型遊戲（高波動 vs 低波動）的正常 RTP 分布差異大，建議以遊戲為單位設定，而非全局統一閾值。

✓ 已決定 · 07 — PM + Customer Service Lead

玩家的分群 Tag 是否對客服後台開放查詢？

已決定：客服後台唯讀可查（查詢條件為玩家帳號 + 遊戲 + 線路），用於玩家異常回報時的還原追蹤；不對玩家前端暴露任何分群資訊。

OPEN · 08 — PM + Data Analyst

A/B Testing 有效樣本的排除範圍如何定義？

目前設計排除兩類：① 白名單特殊 RTP 玩家、② 當月 Tag 切換前的歷史 Spin。
待決：是否也要排除「命中任何非線路預設 RTP 規則的玩家」？排除範圍越大統計越純淨，但有效樣本數會減少、測試週期被迫延長。建議由 Data Analyst 評估各線路的特殊規則覆蓋比例後再決定。

OPEN · 09 — PM + Backend

白名單玩家的 Spin 數據是否完全不寫入 A/B 指標，還是寫入但標記排除？

建議：寫入但標記排除（保留原始數據），讓後台可以彈性調整排除範圍，也方便事後稽核。若完全不寫入，一旦事後需要分析白名單玩家行為就無從追溯。
