01 概要與目的 OVERVIEW & PURPOSE
核心定位 · POSITIONING
抽寶箱第一階段僅派現金獎；第二階段起新增 FREE SPIN 卡這類「非現金獎勵」。本規格定義 FREE SPIN 卡從「後台登記 → 抽中發卡 → 背包持有 → 進遊戲使用 → 整卡結算入帳」的完整細節流程，以及支撐它的獎勵清單 ID 工具與 AWC / 平台串接規格。
v2.1 修訂紀錄（2026/06/22 · 有效期定案 + 啟動機制補強 + 補 UI 視覺稿）
★ 有效期定案：採「發卡 +N 天」相對制、自然日制（發卡日為第 1 天，第 N 天 23:59:59 GMT+8 到期），N 值鎖在卡模板（與遊戲列表一致）。發卡當下換算成絕對 datetime 再送 AWC。原 §10 待確認此項移為「已決」。
★ 啟動機制補強：跳轉啟用由「AWC 進場自動識別」改為「跳轉前先綁定 reference_id 再 launch」——明確指定要啟動哪一張卡，解決「玩家同時持多張對同遊戲有效的卡時無法判斷啟用哪張」與「續玩同一張未用完卡（如 5/10）」的歧義。
★ 補 UI 視覺稿：§04 新增後台獎勵清單工具 wireframe；§05 升級玩家背包前台稿（卡片內展開選遊戲才開始、顯示已用次數 5/10、修正舊 BONUS 顯示為主錢包真金）。

④ 遊戲端 Free Spin 演出的美術製作（屬 Combo 遊戲端 / slot 規格；非 AWC）
演出與數據源權責邊界（修正）：活動由 Combo 主辦，所以局數與獎項結果都是 Combo 提供、遊戲端 Free Spin 演出也由 Combo 渲染。AWC 是「集成商」，只接收 Combo 的結果再通知平台（不是結果的源頭）。資料流向：Combo（局數 + 獎項結果 + 演出）→ AWC（集成 / 接收）→ 平台（入帳 / 通知）。
另一種拓樸：若 Combo 與平台直連（不經 AWC），則由平台與 Combo 串接這段。本平台文件負責「發卡 / 結算 / 串接」，演出美術細節另立 slot 遊戲規格。

三層 ID 關係（campaign_id ↔ reward_id ↔ reference_id）
最容易誤解的一點：「跟著活動」的是 campaign_id，不是 reference_id。reference_id 跟著「每一張發出去的卡」、且必須全域唯一。一個活動發給 1,000 位玩家＝1,000 張卡＝1,000 個不同的 reference_id（同序號重送 AWC 會回 102 序號已使用）。
ID跟著誰數量關係用途
campaign_id活動一個抽寶箱活動 = 1 個來源追溯、對帳分組
reward_id卡型錄（R_FS_3STAR）一款卡 = 1 個，可跨活動複用標記「哪一款卡」
reference_id每一張發出去的卡每張唯一、全域不重複通知 AWC、查詢、防重複
命名慣例（所以它看起來像跟著活動）：習慣把活動代碼當前綴放進 reference_id，但整串必須每張唯一——
reference_id = FS_{campaign_id}_{player_id}_{seq}　例：FS_CHEST0615_user88231_0007
前綴可追溯來源（活動），後段（玩家 + 流水）確保每張卡獨一無二。補發新卡時 campaign_id 可沿用，但 reference_id 必須產新的。

需求 2 · 發卡即鎖定 + 多遊戲切換
一張卡 = 一個共用局數池：玩家用這張卡玩 A 遊戲後，若 rounds_left > 0，可自由切換到列表內 B 遊戲繼續玩，局數從同一池扣。整張卡用完前不做任何中途結算，贏分先累積在卡的暫存上，直到整卡結束才一次給最終獎金。
規則說明
列表鎖定game_ids 在發卡當下決定，遊戲中不可新增 / 移除
自由切換只要 rounds_left > 0 且卡未過期，玩家可在 game_ids 任一遊戲間切換
共用局數A 遊戲花 3 局、B 遊戲花 2 局，總池一致遞減；不分遊戲各自計次
不中途結算切換遊戲、離開遊戲、重新登入都不觸發結算；贏分暫存於卡
結算唯一觸發到達 valid_to（時間結束）統一結算一次
剩餘次數結算時未轉的剩餘局數視同放棄，不補轉、不折現；已累積贏分照常結算給付
結算（v1.1 釐清）：本卡為 Combo 自產，不採 AWC Promotion Tool 的「每轉即時派彩」；於時間結束統一結算 min(total_win, max_conversion)，剩餘未轉次數放棄。

⚠ 防濫用：真金直發的必要補強（取代原「鎖 BONUS」風控）
風險轉移必須處理：舊設計用「高價值卡一律鎖 BONUS、靠平台流水門檻擋提款」來防濫用。改成結算即真金、可立即提領後，這道防線消失。防濫用責任收回到 CRM 中台自持，發卡 / 結算前須通過以下控管，否則就是把提款風險敞開。
控管說明
領取資格限定目標分群 / usertype；非合格玩家不得發卡（防小號刷獎）
頻率 / 額度單人 / 單活動 / 單日的發卡與派彩上限；搭配中台 reserve 鎖額，防 async 空窗超發
黑名單 / 風控旗標命中風控（多帳號、異常提款）即攔截發卡或結算
結算封頂final = min(total_win, max_conversion) 由我方結算層強制套用，杜絕超發
對帳單純化：入帳只有一種目標（主錢包），月結對帳以 campaign_id / ref_free_voucher_id 把每筆中獎歸到正確活動 / 卡片即可，不再需要區分錢包別。
已發卡不可變更 · 擴充走「補發新卡」
已經發到玩家背包、且已 CreateFreeSpin 推給 AWC 的卡，遊戲列表 / 次數等不可原地修改。原因：CreateFreeSpin 是「建立」非「更新」，GameIds 建立當下鎖死、AWC 無 update API；reference_id 不能重用（回 102）。
情境可行性做法
改「卡模板」往後多開遊戲✔ 可行改獎勵清單即可，但僅對「改完之後才發出的卡」生效；已發卡因快照設計維持原樣（避免規則回溯）
幫「玩家手上使用中的卡」加遊戲✘ 技術不可AWC 該筆 free spin 為 immutable，無法追加範圍
真的要讓玩家能玩更多 → 補發新卡（建議）✔ 建議用新 reference_id + 擴充後遊戲列表發一張新卡；玩家持兩張卡＝兩個獨立局數池
作廢舊卡重發✘ 不採用（v1.1）已定調：已發卡不作原地變更 / 作廢，擴充一律走「補發新卡」。無需 AWC 作廢 API。
