01 概要與目的 OVERVIEW & PURPOSE
核心定位 · POSITIONING
抽寶箱第一階段僅派現金獎；第二階段起新增 FREE SPIN 卡這類「非現金獎勵」。本規格定義 FREE SPIN 卡從「後台登記 → 抽中發卡 → 背包持有 → 進遊戲使用 → 整卡結算入帳」的完整細節流程，以及支撐它的獎勵清單 ID 工具與 AWC / 平台串接規格。
v2.0 修訂紀錄（2026/06/17 · 注單模式釐清，取代 BONUS 設計）
★ 結算一律入主錢包真金、可立即提領：經國清向 FC 釐清，整卡結算以一筆 Bet_Amount=0 / Win_Amount=final 注單發放，走既有「遊戲→AWC→平台」注單通道入主錢包，不透過 Promotion Tool、不進 Bonus 錢包、不需 CPS。
★ 移除整套 BONUS 設計：wallet_target（MAIN/BONUS）、bonus_ratio 流水倍數、roll-over、SETTLE 3B（BONUS）、Bonus 入帳 API、Promotion Tool vs Seamless 決策——全部移除。封頂仍由我方結算層自套 min(total_win, max_conversion)。
★ ⚠ 防濫用補強（必做）：原「高價值卡鎖 BONUS」的提款風險防線已不存在；獎勵即真金，領取資格 / 頻率 / 額度 / 黑名單等防濫用須由 CRM 中台自持。
★ 新關卡：逐平台商務開通「任務功能 / Bet=0」＋ RTP 揭露（見 CRM-ROUTE v2.0 / AWC-TOPO v2）。
———（以下為 v1.1 既有紀錄，保留供追溯）———
v1.1 修訂紀錄（2026/06/16 · 依 AWC 串接確認與內部定調）
① 結算規則釐清（本卡為 Combo 自產，非 AWC Promotion Tool 即時派彩版）：於 valid_to（時間結束）統一結算一次——付玩家實際累積贏分，超過 max win 則封頂給 max win（min(total_win, max_conversion)），剩餘未轉次數視同放棄（已累積贏分照付）。多遊戲共用局數池保留。
② 時區改為 GMT+8：有效期 / 過期判定以 GMT+8 為準（原 v1 誤寫 GMT-4；資料 Log 仍以 UTC+0 儲存時間戳）。
③ 對帳以我方自記為主：AWC 不提供報表，對帳真相來源為我方發送紀錄 + callback；AWC 查詢 API（若提供）僅作輔助校驗。
④ 僅支援 Combo 自家遊戲：移除跨遊戲供應商（PG Soft / JILI）差異比較與「作廢已發卡」相關討論（本卡擴充一律走補發新卡）。
🎟️
獎勵清單 ID 工具
後台可登記「非現金獎勵」；第一階段內容只放「FREE SPIN 卡」，每個幣值 × 5 階段 BET，並填入對應幣值的卡片價值。
🔒
發卡即鎖定
發卡瞬間就決定幣別、可用時間、可玩遊戲列表；卡內可自由切換列表遊戲，整卡用完前不結算。
🏦
結算上限（入主錢包真金）
每張卡可指定最終獎金上限值（max_conversion）；結算一律以 Bet=0/Win 注單入主錢包真金，可立即提領（v2.0：不分錢包別、無流水）。

In Scope / Out of Scope
✔ In Scope（本版要做）
① 獎勵清單「非現金 / FREE SPIN 卡」資料結構與後台設定工具
② 抽寶箱抽中 → 發卡 → 背包 → 跳轉 → 結算 全鏈路流程
③ 與 AWC 的 CreateFreeSpin / LoginWithoutRedirect / 查詢 三類串接
④ 最大上限封頂（我方自套 min）+ 結算以 Bet=0/Win 注單入主錢包真金（帶 campaign_id / ref_voucher_id 對帳）
⑤ 背包顯示與一次性跳轉
⑥ 遊戲端 Free Spin 演出由 Combo 渲染（本文件定串接邊界）
✘ Out of Scope（本版不做）
① FREE SPIN 以外的其他非現金獎（如後續的補登入卡、Bonus 卡等）→ 後續階段
② 抽寶箱本身的格子 / 權重 / 解鎖機制（見抽寶箱規格 v3.2）
③ 受限紅利（Bonus 錢包 / 流水 / roll-over）本版不做——獎勵一律真金直入主錢包（v2.0 決策）
④ 遊戲端 Free Spin 演出的美術製作（屬 Combo 遊戲端 / slot 規格；非 AWC）
演出與數據源權責邊界（修正）：活動由 Combo 主辦，所以局數與獎項結果都是 Combo 提供、遊戲端 Free Spin 演出也由 Combo 渲染。AWC 是「集成商」，只接收 Combo 的結果再通知平台（不是結果的源頭）。資料流向：Combo（局數 + 獎項結果 + 演出）→ AWC（集成 / 接收）→ 平台（入帳 / 通知）。
另一種拓樸：若 Combo 與平台直連（不經 AWC），則由平台與 Combo 串接這段。本平台文件負責「發卡 / 結算 / 串接」，演出美術細節另立 slot 遊戲規格。
與抽寶箱規格 v3.3 的關係：抽寶箱已在「獎勵清單」層預留 R_FS_1STAR～5STAR 樁位，並把 FREE SPIN 發放標記為 v2 啟用。v3.3 / v2.0 同步移除了 wallet_target（MAIN｜BONUS）雙錢包——兩份皆改為「獎勵一律入主錢包真金」。本文件為該樁位的深度展開（不重寫抽寶箱本體），命名與其對齊以避免兩份打架。

三層 ID 關係（campaign_id ↔ reward_id ↔ reference_id）
最容易誤解的一點：「跟著活動」的是 campaign_id，不是 reference_id。reference_id 跟著「每一張發出去的卡」、且必須全域唯一。一個活動發給 1,000 位玩家＝1,000 張卡＝1,000 個不同的 reference_id（同序號重送 AWC 會回 102 序號已使用）。
ID跟著誰數量關係用途
campaign_id活動一個抽寶箱活動 = 1 個來源追溯、對帳分組
reward_id卡型錄（R_FS_3STAR）一款卡 = 1 個，可跨活動複用標記「哪一款卡」
reference_id每一張發出去的卡每張唯一、全域不重複通知 AWC、查詢、防重複
命名慣例（所以它看起來像跟著活動）：習慣把活動代碼當前綴放進 reference_id，但整串必須每張唯一——
reference_id = FS_{campaign_id}_{player_id}_{seq}　例：FS_CHEST0615_user88231_0007
前綴可追溯來源（活動），後段（玩家 + 流水）確保每張卡獨一無二。補發新卡時 campaign_id 可沿用，但 reference_id 必須產新的。
獎勵清單 ID 結構（reward_id）
沿用抽寶箱 v3.2 命名，FREE SPIN 卡的 reward_id 以 R_FS_{星級} 表示 5 階段 BET；實際「幣值」由發卡時帶入的 currency 決定，故同一 reward_id 可跨幣值複用，幣值專屬數值（coin_size / face_value / max_conversion）存於子表。

需求 2 · 發卡即鎖定 + 多遊戲切換
一張卡 = 一個共用局數池：玩家用這張卡玩 A 遊戲後，若 rounds_left > 0，可自由切換到列表內 B 遊戲繼續玩，局數從同一池扣。整張卡用完前不做任何中途結算，贏分先累積在卡的暫存上，直到整卡結束才一次給最終獎金。
規則說明
列表鎖定game_ids 在發卡當下決定，遊戲中不可新增 / 移除
自由切換只要 rounds_left > 0 且卡未過期，玩家可在 game_ids 任一遊戲間切換
共用局數A 遊戲花 3 局、B 遊戲花 2 局，總池一致遞減；不分遊戲各自計次
不中途結算切換遊戲、離開遊戲、重新登入都不觸發結算；贏分暫存於卡
結算唯一觸發到達 valid_to（時間結束）統一結算一次
剩餘次數結算時未轉的剩餘局數視同放棄，不補轉、不折現；已累積贏分照常結算給付
結算（v1.1 釐清）：本卡為 Combo 自產，不採 AWC Promotion Tool 的「每轉即時派彩」；於時間結束統一結算 min(total_win, max_conversion)，剩餘未轉次數放棄。

v2.0 釐清：經國清向 FC 釐清，結算不透過 Promotion Tool、不進 Bonus 錢包、不需 CPS。獎勵即真金，搭遊戲本來就在跑的注單流送出，沒有第二條金流管線。原 SETTLE 3B（BONUS）、bonus_ratio 流水、roll-over、Bonus 入帳 API 全部移除。
⚠ 防濫用：真金直發的必要補強（取代原「鎖 BONUS」風控）
風險轉移必須處理：舊設計用「高價值卡一律鎖 BONUS、靠平台流水門檻擋提款」來防濫用。改成結算即真金、可立即提領後，這道防線消失。防濫用責任收回到 CRM 中台自持，發卡 / 結算前須通過以下控管，否則就是把提款風險敞開。
控管說明
領取資格限定目標分群 / usertype；非合格玩家不得發卡（防小號刷獎）
頻率 / 額度單人 / 單活動 / 單日的發卡與派彩上限；搭配中台 reserve 鎖額，防 async 空窗超發
黑名單 / 風控旗標命中風控（多帳號、異常提款）即攔截發卡或結算
結算封頂final = min(total_win, max_conversion) 由我方結算層強制套用，杜絕超發
對帳單純化：入帳只有一種目標（主錢包），月結對帳以 campaign_id / ref_free_voucher_id 把每筆中獎歸到正確活動 / 卡片即可，不再需要區分錢包別。
已發卡不可變更 · 擴充走「補發新卡」
已經發到玩家背包、且已 CreateFreeSpin 推給 AWC 的卡，遊戲列表 / 次數等不可原地修改。原因：CreateFreeSpin 是「建立」非「更新」，GameIds 建立當下鎖死、AWC 無 update API；reference_id 不能重用（回 102）。
情境可行性做法
改「卡模板」往後多開遊戲✔ 可行改獎勵清單即可，但僅對「改完之後才發出的卡」生效；已發卡因快照設計維持原樣（避免規則回溯）
幫「玩家手上使用中的卡」加遊戲✘ 技術不可AWC 該筆 free spin 為 immutable，無法追加範圍
真的要讓玩家能玩更多 → 補發新卡（建議）✔ 建議用新 reference_id + 擴充後遊戲列表發一張新卡；玩家持兩張卡＝兩個獨立局數池
作廢舊卡重發✘ 不採用（v1.1）已定調：已發卡不作原地變更 / 作廢，擴充一律走「補發新卡」。無需 AWC 作廢 API。
04 後台 · 獎勵清單 ID 工具 REWARD LIST TOOL · BACKOFFICE
需求 1 · 對應
在抽寶箱後台的「獎勵清單」新增 非現金獎勵 類別；第一階段內容只開「FREE SPIN 卡」一種，並讓營運針對每個幣值分別設定 5 階段 BET，每階段填入對應幣值的價值（face_value）與發卡參數。
型錄複用：建檔一次 → 活動選用（不是每次新建）
核心觀念：Free Spin 卡是「建檔一次、到處複用的 SKU 型錄」。設抽寶箱活動時，正常只是從現成清單下拉挑卡填進獎項欄位，不需要每次新建一組卡、拿 ID 再填回。

Given卡上限 max_conversion=1,000，玩家整卡 total_win=1,350
When整卡結算
