01概要與目的OVERVIEW
核心主張 · THESIS
把 CRM 設計成「決策大腦（中台）」，把各執行端封裝成可插拔的「渠道適配層 Adapter」。中台只發出統一的發獎/發卡/入帳指令，由 adapter 翻譯成各端 API。Combo 永遠是遊戲商（GP），任何路徑都不持錢包、不結算，只送值；錢包與結算永遠在下游平台商。一套 CRM 即可同時支援直連型平台（MPS / Jackpot Combo）與集成商型渠道（AWC，以注單發獎），新增渠道只需加一個 adapter。
🧠
中台＝決策
玩家分群、活動編排、觸發、campaign 主檔、對帳歸戶。與「執行端」無關，不結算。
🔌
Adapter＝執行
把統一指令翻成各端 API：集成商型走 AWC，直連型走 MPS / Jackpot Combo。
➕
可擴展
新增渠道＝加一個 adapter instance，中台與既有活動邏輯不動。
In Scope / Out of Scope
✔ In Scope
① 三層架構與職責切分；② 統一渠道介面定義；③ 集成商型 / 直連型 adapter 邊界；④ 與 AWC 注單通道 / Seamless 的對應；⑤ 跨端三大前提（玩家識別 / campaign 命名 / 入帳模型）
✘ Out of Scope
① 各活動的玩法細節（抽寶箱、Free Spin 等另立規格）；② 各平台商內部錢包實作；③ CDP / 數據倉儲建置；④ 具體 API 欄位逐一定義（落到各 adapter 規格）
與既有規格的關係：抽寶箱、Free Spin 卡都是「中台底下的能力 / 活動」；本文件定義它們共用的上層骨架。正因兩者已低耦合（reward_id + campaign_id 窄介面），才能各自插進此架構、並依目標平台選走哪一型 adapter。

03CRM 中台職責HUB RESPONSIBILITIES
職責內容為何放中台
玩家分群 / 名單tag、生命週期階段、受眾鎖定、黑白名單跨端共用，不應綁某一端
活動編排 / 觸發解鎖條件、排程、事件觸發（CDP→CRM）、活動模板決策邏輯，與執行端無關
campaign 主檔campaign_id 源頭、reference_id 產生規則來源唯一、跨端一致才能對帳
獎勵型錄現金 / Free Spin 卡 / 未來品項的 SKU 型錄（reward_id）建檔一次、多活動多渠道複用
對帳歸戶 / 稽核把各 adapter 回報的平台結算結果統一歸戶、稽核、月結中台不結算；它是發放真相 + 對平台回報的歸戶稽核者
風控 / 額度 / reserve發放上限、頻率熔斷、預算控管、保留額度 reserve（發放當下鎖額）集中控管避免各端破口、防超發
中台不做的事：不直接呼叫任何供應商 API、不持有錢包、不算流水、不做結算判定、不管遊戲演出——這些全在平台商，經 adapter 下沉，保持中台「純決策」。
⚠ 超發空窗（必處理）：中台的預算/風控與平台的實際結算之間是 async。若不鎖額，連續發放可能在平台回報前就超出預算（通膨破口）。對策＝發放當下在中台先扣「保留額度 reserve」，待平台結算回報後核銷或釋放。
04渠道適配層（Channel Adapter）ADAPTERS
每個 adapter 實作同一組統一介面，對上接中台指令、對下接該端 API。中台呼叫時只指定 adapter 型別（DIRECT 直連 / AGGREGATOR 集成商），由路由依目標平台送到對應 adapter instance。
統一介面（中台 ↔ adapter 契約）
介面用途直連型對應（MPS / Jackpot Combo）集成商型對應（AWC）
grantReward()發現金 / 發卡（一律真金）平台直連發獎 API（送 win 值 / 指定遊戲）以 Bet=0/Win=X 注單發獎 + CreateFreeSpin（發卡）
launchGame()跳轉進遊戲使用獎勵平台直連啟動doLoginAndLaunchGame / LoginWithoutRedirect
settle()送出獎勵結果（平台收注單即結算入主錢包）送 win 值給 MPS/平台錢包，平台結算回報產生 Bet=0/Win=X 注單，帶 campaign_id / ref_voucher_id
queryStatus()查詢進度 / 對帳平台對帳 APIgetTransactionByUpdateDate（注單對帳）
revoke()作廢 / 取消（能力而定）平台取消 API（若提供）— 已發卡不撤，擴充走補發新卡
revoke() 不是保證操作：作廢能力取決於下游平台/集成商是否提供 API，且須按 reward 型別定義語意。Free Spin 已發卡＝best-effort、不保證可撤（呼應 Decision Log #5「已發卡不可變更，擴充走補發新卡」）。介面保留 revoke()，但各 adapter 須明示「支援 / 不支援 / 部分支援」。
集成商型 Adapter 第一階段首發
封裝 AWC 集成商。發獎走既有注單通道：以 Bet=0/Win=X 注單表達獎勵、入主錢包真金，發卡用 CreateFreeSpin、跳轉用 LoginWithoutRedirect、對帳用 getTransactionByUpdateDate。需處理加解密簽章、GMT-4 時區。不對接 Promotion Tool / Bonus 錢包（v1.3 修正）。第一階段主路由（抽寶箱 / Free Spin 規格本就照此撰寫）。
直連型 Adapter 第二階段
封裝直連平台商（MPS、Jackpot Combo、未來其他直連平台）。同一個 shape、多個 instance。送值、無中介，平台自持錢包並結算。第二階段戰略目標，屆時只是新增一個 instance、中台 L1 不動。
05三大前提（adapter 模式成立的基礎）PREREQUISITES
🆔
① 統一玩家識別 基建
CRM 玩家 ID ↔ 各平台（MPS / Jackpot Combo / AWC 背後平台）玩家 ID 的對應表（player mapping）。沒有它，同一玩家在各端無法歸戶。P0 基礎建設，獨立估時（見 §07）。
🏷️
② 統一 campaign / reference
campaign_id / reference_id 跨端命名一致（已設計，見 Free Spin 規格 §02）。promotionId 對映 campaign_id。
🏦
③ 入帳一律主錢包真金
錢包在平台商；中台只負責送出「發 X 真金」，由 adapter 以注單表達。不分 Bonus／不綁流水（v1.3 收斂）。
最該先釘的是 ①：跨端玩家識別沒對齊，後面分群、對帳、發獎全部串不起來。列為架構落地第一個前置任務（P0 基建），先於任何 adapter 開發。
06與現有系統對接邊界INTEGRATION BOUNDARY
AWC ＝ 集成商型 adapter（走既有注單通道，非 Promotion Tool）
v1.3 修正：經國清向 FC 釐清，任務獎勵不透過 Promotion Tool、不進 Bonus 錢包，而是以一筆 Bet_Amount=0 / Win_Amount=X 的正常注單，搭遊戲本來就在跑的「遊戲→AWC→平台」注單通道送出，由平台結算入玩家主錢包真金。集成商型 adapter 要做的不是「對接促銷工具」，而是「讓遊戲產生帶 campaign_id 標記的零下注中獎注單，並回收注單做對帳」。
中台介面AWC 注單通道實作說明
grantReward（現金 / Free Spin）產生 Bet=0/Win=X 注單；發卡用 CreateFreeSpin獎勵即真金，入主錢包
launchGamedoLoginAndLaunchGame / LoginWithoutRedirect卡片跳轉進遊戲
settle（送出獎勵結果）注單即結算依據，帶 campaign_id / ref_voucher_id無 Promotion Tool、無流水
queryStatus / 對帳getTransactionByUpdateDate / GetFreeSpinSendDetail以自有發送紀錄為主，注單校驗
revoke— 已發卡不撤，擴充走補發新卡不需作廢 API
已不需要 Promotion Tool / Bonus 錢包：原 v1.2 設想「對接 AWC Promotion Tool 拿活動錢包 + 流水」的整段作廢。發獎搭既有注單流，沒有第二條金流管線，所以也不需 CPS。剩下的不是技術串接，而是逐平台商務開通「任務功能 / Bet=0 注單」（見 §08 / AWC-TOPO v2）。
抽寶箱 / Free Spin 如何插入：兩者都是中台底下的「活動 / 能力」，透過 grantReward(...) 發放，按目標平台選走直連型或集成商型 adapter。同一張 Free Spin 卡換平台時，卡邏輯不動，只換 adapter。
07建置策略：兩型 adapter 與平台路由TWO ADAPTER TYPES · ROUTING
決策前提 · PREMISE（v1.2 修正）
結算與錢包永遠在平台商（MPS / Jackpot Combo / AWC 背後平台）——Combo 不持有、不自建、只送值。所以這不是「租 vs 自建」、也不是「選誰家錢包」，而是「同時支援兩型 adapter、按目標平台路由」：直連型（MPS / Jackpot Combo，少一層中介、對帳直達）與集成商型（AWC，以注單發獎、覆蓋廣）。第一階段主路由＝集成商型（AWC）首發（最快上線、規格已就位）；MPS 直連型＝第二階段戰略目標。
關鍵事實（v1.3 修正）：集成商型發獎只需產生一筆 Bet=0 / Win=final（已封頂 min(total_win, 上限)） 的注單、帶 campaign_id，由平台結算入主錢包真金。無 Promotion Tool、無 Bonus 錢包、無流水。先前「自建錢包」「對接活動錢包 / 流水」估算皆作廢。

決策（定調）
✅ 第一階段主路由＝集成商型（AWC）首發（注單通道發獎）：抽寶箱 / Free Spin 規格本就照 AWC API 撰寫，獎勵以 Bet=0/Win=X 注單發放、入主錢包真金，不需 Promotion Tool / Bonus 錢包，最快上線。CRM 只送 {campaign_id, win 金額, 指定遊戲}，由 AWC 背後平台商結算。MPS 直連型＝第二階段戰略目標，屆時新增一個 instance、中台不動。守則：本階段把 AWC 嚴格關在 adapter 內，勿讓簽章/口徑滲進中台 L1。
動工前確認（§05 / §08）：① player mapping 基建先就位；② adapter 統一介面先定義；③ AWC 書面確認：place_bet 帶 campaign_id、Bet=0 零押注中獎注單可處理、注單回呼帶回 ref_voucher_id、CreateFreeSpin / LoginWithoutRedirect 規格；④ 逐平台商務開通「任務功能 / Bet=0」。第 ③④ 為第一階段 go/no-go 關卡。（MPS go/no-go 屬第二階段，可提前探詢不阻塞。）
守則：adapter 隔離
架構紅利：因為各平台/集成商的差異都關在 adapter 內，新增或切換渠道時中台（L1）與活動邏輯完全不動，只「新增 / 切換一個 adapter instance」。守住統一介面，任一渠道都「可插可拔」，其他路由不受影響。
